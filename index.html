Overview

How bitonic sort works

Bitonic sort uses a fixed network of comparators rather than runtime decisions. This data-oblivious design makes it ideal for parallel hardware and privacy-preserving applications.

  • Fixed structure – the comparator layout is set before sorting begins.
  • Bitonic sequences rise then fall, or can be rotated to that shape.
  • Parallel-friendly – network stages can run in parallel, unlike branching code.
Flow

Stages of the bitonic sort.

The algorithm first creates opposing sorted halves, then repeatedly compares elements at fixed distances until the whole block is ordered in the target direction.

  1. Build: sort the first half ascending and the second half descending to form a bitonic sequence.
  2. Compare: mirror each element around the midpoint and swap if out of order.
  3. Refine: recursively merge smaller bitonic blocks until the list is fully sorted.
Pseudocode 
1  procedure bitonicSort(lo, n, dir)
2      if n <= 1 return
3      k := n / 2
4      bitonicSort(lo, k, ASC)
5      bitonicSort(lo + k, k, DESC)
6      bitonicMerge(lo, n, dir)

7  procedure bitonicMerge(lo, n, dir)
8      if n <= 1 return
9      k := n / 2
10     for i from lo to lo + n - k - 1
11         compareAndSwap(i, i + k, dir)
12     bitonicMerge(lo, k, dir)
13     bitonicMerge(lo + k, k, dir)
dir chooses ascending or descending order. lo and n define the active segment.
Example

One worked sequence

Example: with the list [3,7,4,8,6,2,1,5], the algorithm sorts the first four numbers up, the last four down and then merges them through a series of compare-swap passes until [1,2,3,4,5,6,7,8] emerges.

Input: [3, 7, 4, 8, 6, 2, 1, 5]
Build: left side trends upward, right side trends downward.
Merge: fixed-distance comparisons move smaller values toward the target side.
Finish: network resolves to [1, 2, 3, 4, 5, 6, 7, 8].
